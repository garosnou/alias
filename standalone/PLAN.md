# План: Alias на сцене (два экрана на одном ПК)

Цель: ведущий управляет игрой на основном мониторе; зрители и объясняющий видят второй экран (проектор/TV) со словом и краткой обратной связью (вспышка фона при оценке ответа). Весь функционал текущего приложения и визуальный стиль сохраняются.

## Работа без интернета (обязательно)

Площадки часто без связи: приложение должно **полностью работать в офлайне** после того как файлы уже на диске ПК.

- **Никаких запросов наружу во время игры**: не подключать CDN (шрифты, скрипты, картинки по URL). В корневом `index.html` сейчас есть `fonts.googleapis.com` — в `standalone` это **убрать**: либо локальные файлы шрифтов (`@font-face` из папки `standalone/fonts/`), либо стек системных шрифтов (`system-ui`, `Segoe UI`, и т.п.) с максимально близким видом.
- **Синхронизация двух окон** — `BroadcastChannel` и обмен через `postMessage` работают только **между процессами на этом же компьютере**, интернет не нужен.
- **Локальный HTTP** (если понадобится вместо `file://`): это **тот же ПК**, петля `127.0.0.1` — внешняя сеть не используется. Подойдёт встроенный в ОС способ отдать папку (например `python -m http.server`) **без установки пакетов из сети**, если Python уже есть; иначе — один раз скопировать портативный статический сервер вместе с билдом (опционально, этап упаковки).
- **Пользовательские слова** — как сейчас: файл с диска, без облака.
- **Проверка**: отключить Wi‑Fi/Ethernet, открыть `host.html` / `display.html` — всё должно грузиться и играть без ошибок в консоли из‑за недоступных URL.

## Текущая база (корень репозитория)

- Vanilla HTML/CSS/JS: `index.html`, `styles.css`, `script.js`, `words.js`
- Режимы: обычная игра, турнир, соревновательный режим
- Настройки, пользовательские пакеты `.txt`, localStorage, клавиатура

## Архитектура «один ПК, два окна»

1. **Два входа в приложение** (в `standalone/`):
   - `host.html` — полный интерфейс ведущего (копия логики навигации и всех режимов).
   - `display.html` — только «публичный» слой: крупное слово, таймер (опционально), пауза/заставка без кнопок оценки.

2. **Синхронизация без сервера**: `BroadcastChannel('alias-standalone')` (один браузер, два окна/вкладки с `file://` или локальный `http://` — для `file://` нужно проверить поддержку; при необходимости минимальный `npx serve` или встроенный скрипт запуска).

3. **События канала** (черновик протокола):
   - `DISPLAY_STATE` — полное состояние для экрана зрителей: текущее слово, остаток времени, фаза (меню / подготовка / игра / пауза / результаты — для публичного экрана чаще только «игра» и «пауза»).
   - `FLASH` — `{ kind: 'correct' | 'wrong' | 'skip' }` — триггер CSS-анимации фона (зелёный / красный / нейтральный или янтарный для «пропуск»).
   - При старте `display` подписывается; `host` при любом изменении слова/таймера/паузы шлёт снимок состояния.

4. **Соответствие кнопок оригиналу и вспышкам**:
   - «Правильно» → зелёная вспышка на дисплее.
   - «Пропустить» → договориться: либо короткая красная/тёмная вспышка «не зачёт», либо отдельный приглушённый цвет (чтобы не путать с «ошибка команды»). При появлении отдельной кнопки «Неправильно» у ведущего — отдельный тип вспышки.

## Два экрана Full HD (1920×1080)

Оба выхода считаем **1920×1080**, соотношение **16:9** — вёрстка и типографика под это, без привязки к «узкому мобильному» как к основному сценарию.

- **Окно ведущего (`host`)**: тот же визуальный стиль, но макет рассчитан на ширину **до 1920px** — центральная колонка может быть шире, чем `max-width: 600px` в оригинале (например `min(720px, 90vw)` или отдельный класс только в standalone), чтобы кнопки и списки не выглядели «крошечной колонкой» на большом мониторе. Высота **1080px**: скролл допустим на экранах настроек/турнира; игровой экран — ключевые элементы в зоне **~900px** по высоте без обязательного скролла.
- **Окно зала (`display`)**: на весь кадр **`100vw` × `100vh`**, без «мобильного» сужения. Слово — **очень крупно** (`clamp()` или `vw`), таймер и подписи читаемы с расстояния. Лёгкий **внутренний отступ 2–4%** от краёв на случай обрезки проектором (overscan).
- **Полноэкранный режим**: по инструкции ведущий и зал открывают окна на своих дисплеях и при необходимости включают **F11** (или кнопка «полный экран» через Fullscreen API только для `display`, по желанию).
- **Масштаб ОС**: не полагаться на 125%/150% Windows для логики — использовать относительные единицы и `clamp`, чтобы при стандартном 100% на FHD всё было предсказуимо.

## Сохранение функционала

- Общий модуль состояния/игры: вынести из `script.js` в `standalone/js/` (постепенно) или подключать скопированный `script.js` + тонкий слой `display-sync.js`, который перехватывает `correctAnswer`, `skipWord`, таймер, `showScreen` и дублирует на канал.
- Минимальный риск: **копия** корневых файлов в `standalone/` + патчи только для синхронизации и второго HTML, чтобы корень не ломать.

## Этапы разработки

| Этап | Содержание |
|------|------------|
| 0 | Скопировать `words.js`, `styles.css`, `favicon.svg`; завести `host.html` ≈ текущий `index.html`; **вырезать CDN**, локальные/системные шрифты; **адаптация host под FHD** (ширина колонки, отступы). |
| 1 | `display.html` + разметка «только слово + таймер» на **весь кадр 1920×1080**; крупная типографика (`clamp`/`vw`), отступы от краёв; стили в духе `styles.css`. |
| 2 | `broadcast-sync.js`: инициализация канала, сериализация состояния, обработка `FLASH`. |
| 3 | Интеграция в игровой цикл: смена слова, тик таймера, пауза, конец раунда, соревновательный отсчёт 3–2–1. |
| 4 | Турнирный режим: на дисплее показывать имя текущего игрока/команды (если есть в UI ведущего), синхронизировать с `tournament-match` / игровым экраном. |
| 5 | Полировка: полноэкранный режим для дисплея (кнопка или F11 в инструкции), защита от рассинхрона (запрос полного state при фокусе окна). |
| 6 | Документация в этом файле или краткий `standalone/README.md`: как открыть два окна, про `http://localhost` при ограничениях `file://`. |

## Риски и решения

- **file:// и BroadcastChannel**: в ряде браузеров окна с `file://` изолированы; тогда локальный статический сервер на одной машине (порт 5173/8080) — один шаг для пользователя.
- **Дублирование кода**: пока приемлемо; позже можно собрать общий пакет из корня через симлинки или общий `js/` в корне (вне scope, если не попросят).

## Критерий готовности MVP

- Ведущий: как сейчас, все режимы работают.
- Второе окно: всегда показывает то же слово, что и у ведущего в раунде; при «Правильно» / «Пропустить» — заметная вспышка фона заданного цвета; таймер совпадает с ведущим с погрешностью ≤ 1 с.
- На **двух Full HD** интерфейс ведущего не «теряется» по центру; зал — заполнение экрана и читаемость слова с расстояния.
